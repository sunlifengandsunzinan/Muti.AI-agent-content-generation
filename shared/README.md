## 设计背景

运营一个抖音账号（"40了，得干点儿啥了"）时面临几个实际问题：

1. **素材管理混乱** — 口述的故事/感悟散落在对话中，每次写脚本都要重新回忆
2. **标题凭感觉** — 没有系统参考赛道爆款，标题质量不稳定
3. **缺少数据反馈** — 账号数据靠手动查看，看不出趋势
4. **创意重复** — 写脚本时不知道哪些素材用过、哪些角度还没开发

为了解决这些问题，设计了一个多Agent协作系统。一开始只是想找个助手帮忙查数据，后来逐步抽象出 Agent 边界、共享数据协议、事件驱动机制，最终形成这个可运行的架构。

## 架构总览

```
👤 用户层
  峰峰 ──→ 决策、给素材、审核产出

🤖 Agent 层（4+1个独立Agent）
  ┌────────────────────────────────────────────┐
  │  Manager（调度中心）                        │
  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌──────┐ │
  │  │素材管家│ │脚本生成│ │标题封面│ │情报站│ │
  │  └────────┘ └────────┘ └────────┘ └──────┘ │
  └────────────────────────────────────────────┘

📁 共享数据层（所有Agent通过文件系统共享）
  data.json  /  materials/  /  scripts/  /  benchmarking/  /  events.json

🌐 外部层
  抖音 douyin.com  ←  Chromium浏览器  ←  情报站采集
  Ollama qwen2.5:7b ─→  情报站本地分析
```

> 架构图：`shared/architecture.html`

## Agent 详解

### Manager — 调度中心

**职责：** 听懂用户意图，分发给正确的子Agent。不碰内容，不做决策，不中转消息。

```
峰峰说"给我写个脚本" → Manager判断 → 找脚本生成器
峰峰说"我给个素材"   → Manager判断 → 找素材管家
峰峰说"看看数据"     → Manager判断 → 找情报站
```

[完整指南 →](skills/douyin-account-agent/agents/manager.md)

### 素材管家 — 素材入口

**职责：** 接收峰峰口述的故事/感悟，两步入库：
1. **原始存档** — 完整保存口述内容到 `shared/materials/`（一字不改）
2. **结构化入库** — 提取核心情感、金句、角度建议，写到 `shared/data.json`

素材经历四个状态：`unused → ready → used → exhausted`

[完整指南 →](skills/douyin-素材管家/agents/sucai-guanjia.md)

### 脚本生成器 — 创意输出

**职责：** 峰峰说"该出脚本了"时，综合分析三样东西出脚本：
- **素材池** — 哪些素材还没用
- **情报站数据** — 赛道最近什么标题在爆
- **标题模板库** — 自家哪些模板跑得好

脚本风格：口语化口播、跑步视角、15-60秒、结尾带评论钩子。

[完整指南 →](skills/douyin-脚本生成器/agents/script-writer.md)

### 标题封面官 — 包装方案

**职责：** 脚本定稿后，输出至少3套标题+封面+标签互补方案。出方案前参考：
- 情报站采集的赛道爆款数据（知道赛道趋势）
- 标题模板库（知道自家什么模板能打）
- 素材情感基调（保持人设一致）

标题和封面互补：标题负责吸引点击，封面验证内容，不重复信息。

[完整指南 →](skills/douyin-标题封面官/agents/title-cover.md)

### 情报站 — 数据大脑

**职责：** 账号数据采集、赛道对标、爆款追踪、每日晨检、自检分析。可以主动干活（不需要等指令）：
- **每日晨检** — 自动扫账号页面，记录数据变化，异常时通知
- **赛道对标** — 搜索"中年跑步"等关键词，采集爆款标题，更新模板库
- **分析类任务** — 走本地 Ollama 模型，不消耗主模型 token

[完整指南 →](skills/douyin-情报站/agents/qingbaozhan.md)

## 协作流程

### 素材上车（用户触发）

```
峰峰："我有一个素材..."
  → 素材管家接收 → 原始存档到 shared/materials/
                   → 结构化写到 data.json.materials.items
                   → "素材已收"
（完事，等待下一步指令）
```

### 创意输出（用户触发）

```
峰峰："该出脚本了"
  → 脚本生成器综合分析：
      ├─ 素材池（哪些素材待用）
      ├─ 对标爆款（赛道趋势）
      └─ 标题模板（自家数据）
  → 写脚本 → 存档到 shared/scripts/
  → 峰峰审核
```

### 包装方案（用户触发）

```
峰峰："出标题吧"
  → 标题封面官读脚本 + 情报站爆款数据
  → 输出3套标题+封面+标签方案
  → 峰峰选定发布
```

### 数据闭环（可自动）

```
情报站每日晨检：
  → 采集账号最新数据
  → 对比昨日，记录变化
  → 更新 data.json
  → 显著波动时通知峰峰
```

## 数据共享协议

所有 Agent 通过文件系统共享数据，不同时写同一个文件。

### 数据结构

```
shared/
├── data.json              # 主数据：账号信息、排名、素材、模板、对标、任务
├── events.json            # 事件队列（未来流水线用）
├── materials/             # 原始素材存档（素材管家维护）
│   ├── material-10-统一思想的人.md
│   └── ...
├── scripts/               # 定稿脚本（脚本生成器维护）
│   ├── 001-草稿纸提分.md
│   └── ...
└── benchmarking/          # 对标爆款库（情报站维护）
    ├── 一个人跑步.md
    ├── 小李跑到中年.md
    ├── 心语与跑步.md
    └── 热血中年老徐.md
```

### 读写权限矩阵

| Agent | 读 | 写 |
|-------|----|----|
| Manager | events.json, data.json | 只读 |
| 素材管家 | data.json.materials | data.json.materials, shared/materials/, events.json |
| 脚本生成器 | data.json 全部 | shared/scripts/, events.json |
| 标题封面官 | shared/scripts/, data.json | 标题模板更新到 data.json, events.json |
| 情报站 | data.json 全部 | data.json（排名/对标/自检）, events.json |

## 运营模式

### 当前：人驱动模式

Agent 不主动跑，峰峰说了才干活。优点：省 token、峰峰掌控节奏。  
适合项目早期，量不大，决策权在人手里。

### 未来：流水线模式

Manager 定期扫 `events.json`，自动调度下一个 Agent。素材入库后自动触发脚本生成，脚本定稿后自动触发标题封面官。适合量大了、流程稳定后开启。

## 快速启动

如果你接替管理这个系统：

1. 读本 README 了解全貌
2. 读每个 Agent 的 agents/ 文件了解职责
3. 看 `shared/events.json` 是否有未完成事件
4. 看 `shared/data.json` 了解账号当前状态
5. 跟峰峰说"我接上了"

## 为什么这样设计

### 为什么用文件系统不用数据库？

- Git 天然版本管理，每个改动都有记录
- AI Agent 读 JSON 比连数据库快得多
- 零依赖，任何时候换环境都能直接跑

### 为什么不分流用 API 通信？

五个 Agent 都跑在同一个 AI 会话上下文里，通过文件共享比 API 调用更直接。文件是共享存储，Agent 是读取+写入的角色，不需要 HTTP 通信。

### 为什么素材要存两份（原始+结构化）？

结构化数据方便程序读取，原始存档保留峰峰的语气和细节。结构化丢失了"怎么说"，但原始存档补全了这个维度。

### 为什么情报站用本地模型？

分析类任务（对标、趋势、自检）不需要大模型的创意能力，本地 7B 模型够用且不消耗 API 额度。创意类任务（写脚本、出标题）则交给主模型。
